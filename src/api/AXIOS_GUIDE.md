// 1. 核心概念：为什么要做这些改进？
// ============================================================

/\*\*

- 问题 1：网络抖动导致请求失败
- 症状：用户网络不稳定，一个简单请求可能失败
- 方案：自动重试 + 指数退避（越来越长的延迟，避免同时重试）
-
- 示例：第 1 次请求失败 -> 等 500ms -> 重试 -> 失败 -> 等 1000ms -> 重试 -> 成功
-
- 问题 2：Token 过期时多个请求同时刷新 token
- 症状：用户 token 过期，同时发出 10 个请求，全部 401，
-         导致同时调用 10 次"刷新 token"接口，浪费流量，刷新混乱
- 方案：使用"刷新队列"，第 1 个请求刷新，其他请求等待 -> 共享新 token -> 继续
-
- 问题 3：不同错误需要不同处理
- 症状：网络错误、业务错误、权限错误都弹一样的提示，用户体验差
- 方案：分类错误，网络错误提示重试，401 提示登录过期，业务错误显示具体原因
  \*/

// 2. 关键改进：逐步分析代码
// ============================================================

/\*\*

- 改进 1：响应数据类型定义（ApiResponse）
-
- 之前：
- const res = await service.get('/api/user')
- // res 是什么？无法确定，可能是 { code, data, msg } 或其他形式
-
- 现在：
- interface ApiResponse<T = unknown> {
-     code: number      // 业务状态码：200 成功，其他为失败
-     data: T           // 真正的数据
-     msg?: string      // 错误信息
- }
-
- const res = await axios.get<ApiResponse<User[]>>('/api/users')
- // res.data.data 是 User[] 类型，IDE 提示完整
-
- 学到的：定义好通信约定，不要 any
  \*/

/\*\*

- 改进 2：请求配置扩展（RetryConfig）
-
- 之前：
- service.get('/api/users')
- // 无法控制重试次数、延迟
-
- 现在：
- interface RetryConfig {
-     retry?: number      // 重试次数，默认 2
-     retryDelay?: number // 重试延迟，默认 500ms
- }
-
- // 可以在请求拦截器里统一设置默认值
- // 也可以单独请求时覆盖
- service.get('/api/critical-data', { retry: 5, retryDelay: 1000 })
-
- 学到的：通过 interface 扩展原生配置，而非 any
  \*/

/\*\*

- 改进 3：Token 刷新队列
-
- 场景：用户 token 过期
-
- 错误的做法（同时刷新 10 次）：
- 请求 1 -> 401 -> 调用 refreshToken() -> 等待响应
- 请求 2 -> 401 -> 调用 refreshToken() -> 等待响应
- 请求 3 -> 401 -> 调用 refreshToken() -> 等待响应
- ...
- 结果：浪费流量，token 混乱
-
- 正确的做法（只刷新 1 次，其他等待）：
- 请求 1 -> 401 -> 锁定刷新队列 -> 调用 refreshToken()
- 请求 2 -> 401 -> 加入队列，等待
- 请求 3 -> 401 -> 加入队列，等待
- refreshToken() 完成 -> 解锁 -> 唤醒队列中的请求 -> 请求 1、2、3 继续
-
- 实现：
- class TokenRefreshQueue {
-     private isRefreshing = false  // 标记是否正在刷新
-     private queue = []            // 等待的请求回调
-
-     start() { this.isRefreshing = true }
-     done() {
-       this.isRefreshing = false
-       this.queue.forEach(cb => cb())  // 唤醒所有等待
-     }
-     cancel() { 此.isRefreshing = false; this.queue = [] }
- }
-
- 学到的：高并发场景下的同步机制设计（这在大厂必考）
  \*/

/\*\*

- 改进 4：响应拦截器的完整流程
-
- 步骤 1：检查业务状态码
- if (data.code === 200) return data.data
- else 显示业务错误
-
- 步骤 2：检查 HTTP 状态码
- 401 -> Token 过期 -> 触发刷新流程
- 其他 -> 检查是否可重试
-
- 步骤 3：判断是否自动重试
- - 不重试的错误：400(参数错误) 401 403(无权限) 404(不存在) 422(验证失败)
- - 可重试的错误：500(服务器错误) 503(服务不可用) 网络超时
- - 重试方式：指数退避（延迟时间 = retryDelay \* 2^(2-retry)）
-
- 学到的：缓存策略、指数退避（防止雪崩）
  \*/

/\*\*

- 改进 5：错误分类与提示
-
- 网络错误（无响应）：
- ElMessage.error('网络连接失败，请检查网络')
-
- 业务错误（code !== 200）：
- ElMessage.error(data.msg) // 显示后端返回的具体原因
-
- 权限错误（401）：
- ElMessage.error('登录过期，请重新登录')
- localStorage.removeItem('geeker-token')
-
- 学到的：用户友好的错误提示，可以显著降低投诉
  \*/

// 5. 进阶话题
// ============================================================

/\*\*

- 话题 1：什么时候应该重试？
- 可重试：网络超时、5xx 错误、幂等操作（GET、DELETE、PUT）
- 不可重试：4xx 错误（参数错误不会因为重试而变对）、非幂等 POST
-
- 话题 2：指数退避的作用
- 如果服务器挂了，所有客户端同时重试 -> 服务器更加过载 -> 更多请求失败 -> 恶性循环
- 用指数退避：每个客户端错开重试时间，给服务器恢复机会
-
- 话题 3：Token 刷新失败怎么办？
- 场景：用户 token 真的过期了（超过 7 天），刷新也失败
- 处理：清除本地 token，强制跳转到登录页，重新登录
-
- 话题 4：如何避免重复提交？
- 场景：用户快速点击"保存"按钮 10 次，产生 10 个 POST 请求
- 方案：在业务层加"防抖"或"节流"；或者在拦截器里检测重复请求
- （这是更高阶的优化，暂时不实装）
